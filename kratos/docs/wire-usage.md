# Kratos 工程中 Wire 的使用说明

这份文档说明当前 `kratos/` 工程里：

1. 为什么使用 Wire
2. 现在的标准组织方式是什么
3. 哪些代码是开发者手写的
4. 哪些代码是 Wire 自动生成的
5. 如何生成
6. 运行时真正执行的是什么

当前工程已经统一成标准 Kratos `ProviderSet` 风格。

---

## 1. Wire 是做什么的

`Wire` 是 Google 的依赖注入代码生成工具。

它不是运行时容器，也不是反射注入。  
它做的是：

1. 在生成阶段分析构造函数依赖
2. 自动生成一份普通 Go 代码
3. 这份生成代码负责把对象按顺序构造出来

所以可以把它理解成：

> 你手写依赖关系，Wire 帮你生成装配代码。

---

## 2. 当前工程的标准组织方式

现在 `sys-api` 和 `sys-rpc` 都已经整理成统一模式：

```text
internal/
├── data/data.go
├── biz/biz.go
├── service/service.go
└── server/server.go
```

每一层都有一个与子包同名的文件，里面定义：

```go
var ProviderSet = wire.NewSet(...)
```

也就是说：

- `internal/data/data.go`：数据层 ProviderSet
- `internal/biz/biz.go`：业务层 ProviderSet
- `internal/service/service.go`：服务层 ProviderSet
- `internal/server/server.go`：服务器层 ProviderSet

这是当前工程采用的规范。

## 4. 哪些代码是开发者手写的

### 4.1 `wire.go` 是手写的

当前 `wire.go` 已经收敛得很干净。

例如 `sys-api`：

```go
func wireApp(cfg *conf.Bootstrap, logger log.Logger) (*kratos.App, func(), error) {
	panic(wire.Build(server.ProviderSet, data.ProviderSet, biz.ProviderSet, service.ProviderSet, newApp))
}
```

例如 `sys-rpc`：

```go
func wireApp(cfg *conf.Bootstrap, logger log.Logger) (*kratos.App, func(), error) {
	panic(wire.Build(server.ProviderSet, data.ProviderSet, biz.ProviderSet, service.ProviderSet, newApp))
}
```

这里是开发者手写的，作用是：

- 告诉 Wire：最终我要一个 `*kratos.App`
- 告诉 Wire：依赖图来自四层 `ProviderSet`

### 4.2 各层 `ProviderSet` 文件是手写的

例如：

- `internal/data/data.go`
- `internal/biz/biz.go`
- `internal/service/service.go`
- `internal/server/server.go`

这些文件里的：

- `ProviderSet`
- `ProvideXxx(...)`
- `NewXxx(...)`

都由开发者手写。

Wire 不会替你发明这些构造函数，它只是使用这些构造函数。

### 4.3 业务构造函数也是手写的

例如：

- `NewAuthRepo`
- `NewAuthUsecase`
- `NewAuthServiceService`
- `NewHTTPServer`
- `NewGRPCServer`
- `NewRegistrar`

这些仍然都是我们自己实现的。

## 5. 哪些代码是自动生成的

### 5.1 `wire_gen.go` 是自动生成的

例如：

- [/Users/guoc/dev/code_go/src/nai-tizi/kratos/application/sys-api/cmd/server/wire_gen.go](/Users/guoc/dev/code_go/src/nai-tizi/kratos/application/sys-api/cmd/server/wire_gen.go)
- [/Users/guoc/dev/code_go/src/nai-tizi/kratos/application/sys-rpc/cmd/server/wire_gen.go](/Users/guoc/dev/code_go/src/nai-tizi/kratos/application/sys-rpc/cmd/server/wire_gen.go)

文件头会明确写着：

```go
// Code generated by Wire. DO NOT EDIT.
```

所以：

- `wire_gen.go` 不应该手改
- 你改了之后，下次执行 `make wire` 会被覆盖

### 5.2 自动生成的内容是什么

本质上就是普通 Go 代码。

它会把四层 ProviderSet 展开成具体构造顺序，例如：

- 先从 `cfg` 派生子配置
- 创建 data 层对象
- 创建 biz 层对象
- 创建 service 层对象
- 创建 server
- 最后创建 `kratos.App`

所以 `wire_gen.go` 不是配置文件，它就是一份自动写出来的装配代码。

## 6. 当前工程里，Wire 怎么组织依赖

### sys-api

当前依赖图可以理解成：

```text
cfg + logger
  -> data.ProviderSet
  -> biz.ProviderSet
  -> service.ProviderSet
  -> server.ProviderSet
  -> newApp
  -> *kratos.App
```

#### data.ProviderSet

负责提供：

- `conf.RPC`
- `*RPCClientSet`
- 各个 repo

例如：

- `ProvideRPCConfig`
- `NewData`
- `NewAuthRepo`
- `NewUserRepo`

#### biz.ProviderSet

负责提供：

- 各个 usecase

例如：

- `NewAuthUsecase`
- `NewUserUsecase`
- `NewAttachmentUsecase`

#### service.ProviderSet

负责提供：

- 各个 HTTP service 实现

例如：

- `NewAuthServiceService`
- `NewUserServiceService`

#### server.ProviderSet

负责提供：

- `conf.HTTP`
- `*WebSocketHub`
- `*GatewayDeps`
- `*http.Server`

例如：

- `ProvideHTTPConfig`
- `NewWebSocketHubProvider`
- `ProvideAuthGateway`
- `NewGatewayDeps`
- `NewHTTPServer`

### sys-rpc

当前依赖图可以理解成：

```text
cfg + logger
  -> data.ProviderSet
  -> biz.ProviderSet
  -> service.ProviderSet
  -> server.ProviderSet
  -> newApp
  -> *kratos.App
```

#### data.ProviderSet

负责提供：

- `conf.Data`
- `conf.Auth`
- `conf.JWT`
- `conf.Observability`
- `*Resources`

例如：

- `ProvideDataConfig`
- `ProvideAuthConfig`
- `ProvideJWTConfig`
- `ProvideObservabilityConfig`
- `NewData`

#### biz.ProviderSet

负责提供：

- 各个 usecase

例如：

- `NewAuthUsecase`
- `NewMenuUsecase`
- `NewLoginLogUsecase`
- `NewOperLogUsecase`

#### service.ProviderSet

负责提供：

- 各个 gRPC service 实现

例如：

- `NewAuthServiceService`
- `NewMenuServiceService`
- `NewLoginLogServiceService`

#### server.ProviderSet

负责提供：

- `conf.GRPC`
- `*grpc.Server`
- 可选 `registry.Registrar`

例如：

- `ProvideGRPCConfig`
- `NewGRPCServer`
- `NewRegistrar`

## 7. `wire.go` 和 `wire_gen.go` 的关系

可以这样理解：

### `wire.go`

是“装配声明”。

你在这里写：

- 顶层依赖入口
- `wire.Build(...)`

### `wire_gen.go`

是“装配结果”。

Wire 根据 `ProviderSet` 和构造函数，把依赖关系展开成实际 Go 代码。

也就是说：

- `wire.go` 决定结构
- `wire_gen.go` 体现结果

---

## 8. 运行时真正执行的是哪份代码

运行时真正执行的是：

- `wire_gen.go`

不是 `wire.go`。

`wire.go` 在生成阶段用来喂给 Wire，  
生成完成后，真正参与编译和运行的是 `wire_gen.go`。

这也是为什么：

- 你不能手改 `wire_gen.go`
- 你要改依赖关系，应该去改 `wire.go` 和各层 `ProviderSet`

---

## 9. main.go 还负责什么

当前两个服务的 `main.go` 都很薄。

主要只做四件事：

1. 读取配置
2. 创建 logger
3. 调 `wireApp(cfg, logger)`
4. 调 `app.Run()`

也就是说，复杂的对象创建已经不放在 `main.go` 里了。

## 10. 如何生成 Wire 代码

执行：

```bash
make wire
```

它会运行：

```bash
go run github.com/google/wire/cmd/wire ./application/sys-api/cmd/server ./application/sys-rpc/cmd/server
```

然后更新：

- `application/sys-api/cmd/server/wire_gen.go`
- `application/sys-rpc/cmd/server/wire_gen.go`

---

## 11. 什么时候需要重新执行 `make wire`

当你修改了依赖装配关系时，就需要重新执行。

典型场景：

1. 新增了一个 provider 函数
2. 某层 `ProviderSet` 增删了 provider
3. 某个构造函数参数变了
4. `data / biz / service / server` 某层的依赖结构变了
5. 新增了新的 server 依赖或 cleanup 链

如果只是改业务方法内部逻辑，比如：

- 登录校验细节
- repo 查询细节

通常不需要重新跑 `make wire`

因为依赖图没变。

- `cmd/server/wire.go`
- `internal/data/data.go`
- `internal/biz/biz.go`
- `internal/service/service.go`

### 不要手改

- `wire_gen.go`

因为它始终应该由 `make wire` 生成。

## 13. 为什么当前这种方式更规范

相比以前把：

- `Repositories`
- `Usecases`
- `Services`

作为聚合入口再由 `wire.go` 直接拼装，

现在这套方式更符合 Kratos 常见风格，因为：

1. 每层职责更清楚
2. 每层依赖入口固定
3. `cmd/server/wire.go` 足够简洁
4. 后续新增 provider 时，只需要去对应层补 `ProviderSet`
5. 更接近标准 `kratos-layout` 的阅读习惯

## 14. 推荐阅读顺序

如果你后面要继续读这套装配逻辑，最推荐的顺序是：

1. `main.go`
2. `cmd/server/wire.go`
3. `cmd/server/wire_gen.go`
4. `internal/data/data.go`
5. `internal/biz/biz.go`
6. `internal/service/service.go`
7. `internal/server/server.go`

按这个顺序，你会很容易看出：

- 配置从哪里进入
- 哪些对象在哪一层创建
- 最终如何组装成一个 Kratos App

---

## 15. 一句话总结

当前工程已经切成标准 Kratos Wire 风格：

- `wire.go`：手写，声明顶层装配关系
- `internal/*/*.go`：手写，定义各层 `ProviderSet`
- `wire_gen.go`：自动生成，真正负责把对象按顺序组装出来
- `main.go`：只负责读取配置、创建 logger、启动 app

所以现在这套结构是：

> 各层自己声明 provider，顶层只负责组合，生成代码负责真正装配。
